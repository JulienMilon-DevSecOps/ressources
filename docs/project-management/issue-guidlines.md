# 🧩 Issue Guidelines

## 🎯 Objectif
Standardiser la création des issues afin d'améliorer la clarté, la collaboration et la qualité du projet.

---

## 📌 Quand créer une issue ?

Créer une issue pour :
- 🐛 Signaler un bug
- ✨ Proposer une fonctionnalité
- 📚 Ajouter ou améliorer la documentation
- 🔧 Effectuer une tâche technique

---

## 🧱 Structure d’une bonne issue

### 1. Titre
Clair et concis

Exemples :
- `[API] Ajouter endpoint login`
- `[BUG] Erreur 500 sur /users`

---

### 2. Description

Inclure :
- Contexte
- Problème ou besoin
- Objectif

---

### 3. Détails techniques

- Stack concernée
- Fichiers impactés
- Contraintes

---

### 4. Critères d’acceptation

Checklist claire :

- [ ] Fonctionnalité implémentée
- [ ] Tests OK
- [ ] Documentation mise à jour

---

## 🏷️ Types d’issues

- `bug`
- `feature`
- `enhancement`
- `documentation`
- `chore`

---

## ⚠️ Bonnes pratiques

- Une issue = un sujet
- Être précis
- Ajouter du contexte
- Utiliser les templates de ce projet ou autre (/templates/issues)

---

## ❌ À éviter

- Issues vagues
- Plusieurs sujets dans une issue
- Absence de critères d’acceptation
