# 🤝 Contributing Guide

Ce projet de base est un projet personnel pour mettre un ensemble de ressources 
Merci de vouloir contribuer à ce projet ! 🎉
Ce document décrit les règles et bonnes pratiques à suivre pour garantir la qualité et la cohérence du repository.

---

## 📌 Objectif du projet

Ce projet de base est un projet personnel qui vise à fournir un ensemble de ressources autour de :

* Développement API
* Frontend
* CI/CD
* Bonnes pratiques de développement

Il est rendu public pour partager les ressources et voir des proposition d'amélioration et d'idée venu d'autre personne et contributeur qui le souhaiterait. Merci de bien respecter le travail réaliser. Actuellement les branches principales main et develope sont protégées.

---

## 🚀 Comment contribuer

### 1. Fork du repository

Crée un fork du projet puis clone-le localement :

```bash
git clone https://github.com/<your-username>/<repo>.git
cd <repo>
```

---

### 2. Créer une branche

Ne travaille **jamais directement sur `main`** ou `develop`.

```bash
git checkout -b feature/nom-de-la-feature
```

Vous pouvez même au préalable créer une Issue (utiliser aussi le template) et créez une branche à partir de cette Issue et de la branche develop. 

---

### 3. Développer

* Respecter les conventions du projet
* Ajouter de la documentation si nécessaire
* Ajouter des exemples concrets si pertinent

---

### 4. Commit

Utiliser une convention de commits claire :

```bash
feat: ajout d’un template API
fix: correction du README
docs: ajout documentation CI/CD
chore: mise à jour des dépendances
```

---

### 5. Push & Pull Request

```bash
git push origin feature/nom-de-la-feature
```

Puis :

* Ouvrir une Pull Request (PR)
* Remplir le template
* Décrire clairement les changements

---

## 🔀 Process de validation

Toutes les contributions passent par :

* ✅ Une Pull Request
* ✅ Une revue de code
* ✅ Une validation par un mainteneur

Aucun merge direct sur `main` n’est autorisé.

---

## 📋 Règles de Pull Request

Une PR doit :

* Être claire et concise
* Avoir un titre explicite
* Décrire le contexte et les changements
* Être de taille raisonnable

---

## 🧪 Qualité du code

Avant de soumettre une PR :

* Vérifier que le code fonctionne
* Ajouter ou mettre à jour les tests si nécessaire
* Vérifier la lisibilité et la structure

---

## 📚 Documentation

Toute nouvelle fonctionnalité ou modification doit :

* Être documentée dans `docs/`
* Être compréhensible par quelqu’un d’externe

---

## 🧩 Types de contributions

Vous pouvez contribuer de plusieurs façons :

* 🐛 Correction de bugs
* ✨ Ajout de fonctionnalités
* 📚 Documentation
* 🧪 Ajout de tests
* 🧹 Refactoring

---

## ⚠️ Bonnes pratiques

* Faire des PR petites et ciblées
* Éviter les changements non liés
* Expliquer les choix techniques
* Être respectueux dans les échanges

---

## 🧑‍⚖️ Code Review

Les reviewers peuvent :

* Demander des modifications
* Poser des questions
* Suggérer des améliorations

👉 Une PR peut être refusée si elle ne respecte pas les standards.

---

## ❓ Questions

Si vous avez une question :

* Ouvrez une issue
* Ou discutez dans les PR

---

## 🙌 Merci

Merci pour votre contribution ! 🚀
